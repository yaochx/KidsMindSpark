# UI 规则

## 总原则

- 面向小学生，界面应清晰、温和、可理解。
- 不做纯聊天式 UI。
- 图形化优先，尤其是故事主线确认。
- 不堆叠长正文，长内容必须拆成卡片、节点、分页或分镜。
- 正文、卡片、节点、按钮、浮层不得重叠。

## 字号与排版

- 页面主标题建议 28-36px。
- 区块标题建议 20-24px。
- 正文建议 16-18px。
- 儿童阅读内容建议行高不低于 1.6。
- 不使用过小、过密、低对比度文本。

## 页面最大宽度

- 主内容最大宽度建议 1120-1280px。
- 表单页最大宽度建议 720-880px。
- 漫画预览可使用更宽容器，但单页预览区域必须保持明确边界。
- 移动端必须使用单列或可横向滚动的图形画布。

## 卡片规则

- 卡片用于承载一个明确的信息单元。
- 卡片间距建议不小于 16px。
- 卡片内部 padding 建议不小于 16px。
- 不把大段正文塞入单张卡片。
- 不允许卡片嵌套卡片形成复杂视觉噪音。

## 主线节点规则

- 节点卡片建议宽度 180-240px。
- 节点卡片建议最小高度 96px。
- 节点间水平或垂直间距不小于 32px。
- 节点标题必须短。
- 节点描述最多 2-3 行，超出使用详情面板。
- 节点图必须有滚动、缩放或自动布局策略，避免节点互相覆盖。

## 按钮规范

- 主要动作按钮只保留当前阶段的关键动作。
- 主按钮文案必须明确，例如“生成主线”“确认主线”“生成 32 页分镜”“导出 PDF”。
- 危险或重置操作必须弱化并二次确认。
- 按钮最小高度建议 40px。
- 移动端按钮宽度应便于点击。

## 空状态

- 空状态必须告诉用户下一步可做什么。
- 空状态不显示技术错误栈。
- 示例：
  - 尚未生成主线：显示“输入故事概念后，可以生成故事主线。”
  - 尚未生成分镜：显示“确认主线后，将生成固定 32 页漫画分镜。”

## Loading 状态

- 所有生成动作必须有 loading 状态。
- loading 文案应短，不出现长段解释。
- loading 时禁止重复提交同一生成请求。
- 可显示阶段信息，例如“正在整理主角和目标”“正在安排 32 页分镜”。

## 错误状态

- 错误提示应儿童友好、可恢复。
- 技术细节可放入开发调试区域，不直接展示给小学生。
- 生成失败时保留用户已输入内容。
- 主线或脚本生成失败不得产生半成品覆盖已确认数据。

## 漫画预览规则

- 每页显示页码。
- 每个分镜必须有图像区域。
- 对白气泡不应遮挡关键画面。
- 每页对白必须短。
- mock 图像占位也要表现为彩色漫画图像区域，而不是纯灰框。

## 响应式要求

- 桌面端优先使用宽布局和可视化流程。
- 移动端允许主线图横向滚动。
- 所有文本必须在容器内换行。
- 不使用固定高度承载不定长正文，除非配合滚动。
